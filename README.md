# synergy
synergy practice

### 1. Атрибуты (Свойства)
В классе `WORKER` были определены следующие атрибуты, отражающие ключевые характеристики сотрудника организации (на примере Университета «Синергия»):
   `__fio` (str): Фамилия и инициалы работника. Используется для идентификации личности.
   `__position` (str): Название занимаемой должности (например, «Преподаватель», «Менеджер по продажам», «Системный администратор»).
   `__salary` (float/int): Размер заработной платы. Важный финансовый показатель.
   `__year_start` (int): Год поступления на работу. Необходим для расчета стажа.

Примечание: Опыт работы (стаж) не хранится как отдельный статический атрибут, так как он является производной величиной, зависящей от текущей даты и года поступления. Он вычисляется динамически методом `get_experience()`.

### 2. Методы (Функции)
Для управления объектами класса реализован следующий набор методов:
   Конструктор `__init__`: Инициализирует объект, присваивая начальные значения атрибутам. Поддерживает значения по умолчанию.
   Геттеры и Сеттеры (`@property`):
       Позволяют безопасно читать значения атрибутов.
       Позволяют изменять значения с предварительной валидацией (проверкой типов данных и логических ограничений, например, зарплата не может быть отрицательной).
   `display()`: Метод для форматированного вывода информации о работнике в консоль.
   `get_experience()`: Метод бизнес-логики, который вычисляет текущий стаж работника в годах, сравнивая текущий год с годом трудоустройства.
   Деструктор `__del__`: Выполняет очистку ресурсов (в учебном контексте — выводит сообщение об удалении объекта), демонстрируя жизненный цикл объекта.

### 3. Абстракция
Класс `WORKER` представляет собой абстракцию реального понятия «Сотрудник».
   Мы отвлеклись от несущественных деталей (цвет глаз, хобби, домашний адрес), оставив только те данные, которые необходимы для решения поставленной задачи (учет стажа и вывод списка).
   Абстракция позволяет работать с объектом как с единым целым, скрывая внутреннюю сложность хранения данных и логику расчета стажа.

### 4. Инкапсуляция
В реализации соблюдены принципы инкапсуляции:
   Все атрибуты объявлены как приватные (с префиксом `__` в Python, что вызывает name mangling — искажение имени). Это предотвращает прямой доступ к данным извне класса (например, нельзя написать `worker.__salary = -100`).
   Доступ к данным осуществляется исключительно через публичные методы-аксессоры (геттеры и сеттеры), decorated с помощью `@property`.
   В сеттерах реализована защита от некорректных данных (валидация), что гарантирует целостность состояния объекта.

### 5. Наследование
В текущей версии задачи класс `WORKER` является самостоятельным и не наследуется от других пользовательских классов.
   Потенциал расширения: При необходимости класс `WORKER` мог бы наследоваться от базового класса `PERSON` (Человек), который содержал бы общие поля (ФИО, дата рождения), чтобы избежать дублирования кода, если бы в системе учитывались также студенты или абитуриенты. Однако в рамках текущего ТЗ это избыточно.

### 6. Полиморфизм
В данной задаче полиморфизм не использован явно, так как нет иерархии подклассов.
   Возможность применения: Если бы потребовалось различать поведение сотрудников разных отделов (например, у преподавателей есть метод `conduct_lecture()`, а у менеджеров `make_call()`), можно было бы создать подклассы `Teacher(WORKER)` и `Manager(WORKER)`, переопределив в них метод `display()` или добавив специфичные методы. В текущем решении все объекты `WORKER` ведут себя единообразно.

### 8. Тестирование
Для проверки корректности работы класса были проведены следующие тесты (вручную и через запуск `main.py`):
1.  Тест создания объекта: Успешное создание объекта с валидными данными.
2.  Тест валидации: Попытка установить отрицательную зарплату или некорректный год вызвала `ValueError`, что подтверждает работу защитных механизмов.
3.  Тест расчета стажа: Для сотрудника, принятого в 2020 году, при текущем годе 2026 метод `get_experience()` вернул 6 лет.
4.  Тест фильтрации: При запросе стажа > 5 лет сотрудник со стажем 6 лет был выведен, а со стажем 2 года — нет.
5.  Тест пустого результата: При запросе несуществующего большого стажа (например, 100 лет) вывелось сообщение «Сотрудников с указанным стажем не найдено».

### 9. Итерация и возможности улучшения
После написания кода был проведен рефакторинг и анализ возможных улучшений:
1.  Хранение данных: Сейчас данные хранятся в оперативной памяти (список). Для реальной системы необходимо добавить сохранение в базу данных (SQLite/PostgreSQL) или файл (JSON/CSV), чтобы данные не терялись после перезапуска программы.
2.  Интерфейс: Консольный интерфейс удобен для отладки, но для пользователей университета лучше подошел бы графический интерфейс (GUI) на Tkinter/PyQt или веб-интерфейс.
3.  Логирование: Вместо `print` в деструкторе и при ошибках лучше использовать модуль `logging` для записи событий в файл журнала.
4.  Типизация: Можно добавить аннотации типов (Type Hints) для большей строгости кода (например, `def __init__(self, fio: str, ...)`).
